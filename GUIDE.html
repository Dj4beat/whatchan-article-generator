2. How It Works

The generator is a web page that runs in your browser. You open it like any website — no software to install. Behind the scenes, it connects to several services to do the heavy lifting:

  • GitHub Pages — hosts the web page for free. When we update the code and push it to GitHub, the live site updates automatically in about 2 minutes.
  • Cloudflare Worker — a small program that runs on Cloudflare’s servers. It acts as a middleman between the browser and the AI services. This is necessary because (a) it keeps the API key secret (the browser can’t hold secrets safely) and (b) it fetches RSS feeds and web pages that browsers aren’t allowed to fetch directly due to security restrictions (called CORS).
  • OpenRouter — a service that provides access to 20+ AI models (Google Gemini, OpenAI GPT, Anthropic Claude, etc.) through a single account. We pay per use — a few pence per article for most models.
  • Perplexity Sonar Pro — a special AI model that can search the live web in real time. We use it to verify current facts (who manages a club right now, what the latest result was) before the article is written.
  • Cloudflare KV — a permanent storage system on Cloudflare’s servers. We use it to store editorial feedback and learning data. Unlike browser storage, this data survives clearing your browser, and is accessible from any device.

IN PLAIN ENGLISH You click “Generate” in your browser → the browser sends the request to the Cloudflare Worker → the Worker asks the AI model to write the article → the article comes back to your browser, fully formatted and ready to paste into WordPress.

3. Using the Generator

News Feed tab (default)

Shows live headlines from BBC Sport, Sky Sports, The Athletic, and The Guardian. Click any headline to expand it, select a category, and click Generate WhatChan Article. You can also paste a URL directly or upload a PDF/DOCX/TXT as extra context.

Evergreen tab

Pre-built topics that are always relevant (e.g. "Premier League Top Scorer 2026"). Select one or more, set batch size, and generate. The Topical Article Builder at the top lets you pick a team + type (Transfer News, Match Preview, etc.) for timely articles.

Generated tab

All generated articles appear here with badges, action buttons, and collapsible panels. Articles persist for 7 days in your browser. Each card shows: category badge, model badge (purple), article length, QC verdict, and review status.

4. Custom Topic & Club Templates

The Custom Topic card in the News Feed tab lets you write about anything. When Football is selected as the sport:

  • A Club / Team dropdown appears (grouped by league: PL, Championship, League One, etc.)
  • A URL Context field appears for pasting match report or stats page URLs
  • Selecting a category auto-fills an expert journalist template into the description box

14 TEMPLATES One for every football category, written from the perspective of an Editor, Football Expert, Writer, and QC Agent. Each covers every section a sports journalist would be expected to include. Templates use [CLUB] placeholder replaced with your selected club name.

You can always edit, delete, or add to the template text. If you manually type into the description, changing the category will not overwrite your edits.

QUALITY RULES Every sentence must contain a specific fact. Generic filler like "both teams will be looking to" is banned. If a section has no real data, it is omitted entirely. Pullquotes must be real quotes from real people, never fabricated.

5b. Match Infographics

You can add a visual statistics infographic to any article. It shows comparison bars for each stat (possession, shots, xG, corners, etc.) with the home team in blue and the away team in red.

How to add an infographic

  1. Tick the Include Infographics checkbox (below the White Label bar)
  2. Paste a BBC Sport match page URL into the stats URL field — for example: https://www.bbc.com/sport/football/live/cx2rwjnpwe5t#MatchStats
  3. Click Fetch Stats
  4. You should see a green confirmation showing the teams, score, and number of stats — for example: “✓ Atletico Madrid 1-2 Barcelona (10 stats)”
  5. Now generate your article as normal — the infographic will be inserted automatically at about 40% through the article

ZERO COST The stats are extracted directly from the BBC page’s embedded data — no AI tokens are used. The data comes from BBC’s own __INITIAL_DATA__ JSON which is embedded in every match page. The generator reads this structured data and builds the visual bars in JavaScript — it’s instant, free, and always shows the correct match.

What stats are extracted

The system extracts every available stat from the BBC page, typically including: Possession, Shots, Shots on Target, Expected Goals (xG), Corners, Fouls, Passes, Pass Accuracy, Tackles, Saves, and Offsides.

What it looks like

The infographic appears as a dark navy block in the article with:

  • A gold “MATCH STATISTICS” header
  • Team names and score
  • Horizontal comparison bars for each stat — blue (home) vs red (away), with proportional widths calculated from the actual numbers
  • Stat labels centred below each bar

BBC SPORT ONLY Currently only BBC Sport match page URLs are supported. The URL must be a BBC Sport “live” page (e.g. /sport/football/live/...). Other stats sites (WhoScored, ESPN) are not yet supported for infographics.

6. Choosing a Model

The dropdown in the header lets you pick from 20+ models across Google, OpenAI, Anthropic, Mistral, xAI, Meta, DeepSeek, and Perplexity. The model you select is used for the main writing step and E-E-A-T polish. Analysis, QC, and hallucination scanning always use Gemini 2.5 Flash (cost-efficient).

Each generated article shows a purple model badge so you know which model wrote it. Feedback is tied to the specific model, so the learning system knows which model made which mistakes.

7. The Generation Pipeline

Standard articles (News Feed / URL / Evergreen)

1
Verify — Perplexity Sonar Pro searches the web for current facts (managers, results, standings).
2
Analyse — Gemini extracts key facts from the source. Web-verified facts marked [VERIFIED].
3
Write — Your selected model writes the full article using only verified facts as ground truth.
4
Hallucination scan — Compares draft against source. Strips fabricated facts, fake quotes, generic filler. Findings saved to feedback memory.
5
QC + SEO — Source-traceability check with auto-retry loop (up to 3 attempts). SEO fields generated.
6
E-E-A-T polish — Experience, Expertise, Authoritativeness, Trustworthiness enhancement.

Match articles (Match Preview / Match Report)

Same as above but preceded by deep research: Perplexity fixture research + reference site scraping + stats research. All run before the writing step.

8. Quality Controls & Auto-Retry

AUTO-RETRY If QC fails, the article is automatically rewritten with all accumulated issues and QC'd again — up to 3 total attempts. QC-FAIL articles only reach you if all 3 attempts are exhausted.

9. Feedback & Learning System

The generator learns from every mistake. Feedback is stored permanently in Cloudflare KV — it survives browser clears, works across devices, and compounds over months.

What triggers learning

EventSource typeSaved automatically?
Hallucination scan strips contentauto-hallucinationYes
QC requires retries to passauto-qc-correctedYes
QC exhausts all 3 attemptsauto-qc-exhaustedYes
Editor clicks Feedback buttonhumanYes, on submit

How to give feedback

Click the purple Feedback button on any article card. Describe what needs fixing in the textarea and click Reprocess Article. The article goes through the full pipeline again with your feedback injected, and updates in place.

Model-aware learning

Each feedback entry records which model generated the article. On future articles:

  • Same model sees its own feedback as "YOUR DIRECT FEEDBACK" (harshest framing)
  • Different model sees it as "SHARED KNOWLEDGE (from [model name])" (advisory)
  • Repeating a known mistake after being warned is treated as worse than the original error

11. Saved Articles

Articles save to your browser automatically and persist for 7 days. They survive page refreshes and browser restarts. Cards show "saved today", "saved yesterday", or "saved 3d ago".

Click Delete on any card to remove it. Articles are per-browser, per-domain (different browsers have separate article lists, but feedback memory is shared via Cloudflare KV).

12. White Label Mode

Tick the White Label checkbox below the model selector. Enter a domain name and optionally an author name. Effects:

  • Schema uses your domain instead of whatchan.co.uk
  • Author credit uses your name (or no byline if left blank)
  • Both banners (sport TV + blog) are suppressed
  • No WhatChan mentions anywhere in the article
  • Blue White Label badge on the card

13. DOCX Export

Click DOCX on any article card to download a Word document. The export strips all code (schema, SEO comments, banners, tags) and produces clean formatted text suitable for editing in Word or Google Docs.

15. Security

The main thing to protect is the OpenRouter API key — this is what lets the system call AI models, and each call costs money. Here’s how it’s protected:

  • The API key is stored on Cloudflare’s servers, not in the web page. Nobody can see it by viewing the page source or inspecting the code.
  • A spend cap on OpenRouter limits how much can be spent even if someone found a way to abuse it. Set this at openrouter.ai/settings/limits — £10/month is recommended.
  • A token (WC_TOKEN) is included in every request from the browser to the Worker. If someone tries to call the Worker directly (e.g. from their own script), they need this token. It’s visible in the page source, but combined with the spend cap, the risk is minimal.
  • CORS restrictions mean only the official WhatChan site (and localhost for testing) can talk to the Worker. Other websites cannot send requests to it even if they know the URL.
  • Feedback data in KV is only readable and writable through the Worker API, which requires the token. Nobody can access it directly.

IMPORTANT The code is in a public GitHub repository (required for free GitHub Pages hosting). This means anyone can read the HTML source code. This is fine — the only secret (the API key) is stored on Cloudflare, not in the code. The spend cap is your safety net.

16. Deploying Changes

Frontend (the website — GitHub Pages)

The live website is just the file docs/index.html hosted on GitHub. When you update this file and push it to GitHub, the live site updates automatically in about 2 minutes. Here are the commands:

git add docs/index.html          # Stage the changed file (tell git "I want to save this change")
git commit -m "Description"      # Save the change with a message describing what you did
git push                         # Upload the saved change to GitHub — the live site updates automatically

Always test after pushing: Open the site in an incognito/private window (Ctrl+Shift+N in Chrome) to make sure you see the latest version. Normal browser windows sometimes show a cached (old) version.

Worker (the Cloudflare backend)

The Worker is the middleman that handles API calls, RSS feeds, and feedback storage. If you change any files in the worker/ folder, you need to redeploy it:

cd worker                        # Navigate into the worker folder
npx wrangler deploy              # Upload the updated code to Cloudflare

This takes about 10 seconds. The Worker updates instantly — no waiting for a rebuild.

WHEN DO YOU NEED TO REDEPLOY THE WORKER? Only when files inside the worker/ folder change. If you only changed docs/index.html (the website), you do NOT need to touch the worker — just git push is enough.

Rolling back to a previous version

Every change is tagged in git with a version number. You can roll back to any version at any time.

CURRENT VERSIONS

Step 1: See all available versions

This lists every tagged version. Think of tags as bookmarks on specific points in the project’s history.

git tag -l

Step 2: See what changed between versions

This shows a short summary of every change made between two versions, so you know what you’d be undoing.

git log v2.0..v2.1 --oneline

Step 3: Roll back (safe — keeps your full history)

This undoes all changes made after v2.0, but keeps the full history intact. You can always roll forward again later if you change your mind. Copy and paste these three commands one after another:

git revert v2.0..HEAD --no-commit
git commit -m "Revert to v2.0"
git push

What happens: The live site will update in ~2 minutes to look exactly like it did at version 2.0. All changes made since then are removed from the live site, but the code is still in the git history if you ever need it back.

Step 4: Redeploy the worker (only if the version change affects the worker)

If the version you’re rolling back from changed any worker files (check the CHANGELOG), you need to redeploy:

cd worker
npx wrangler deploy

If you’re unsure whether the worker needs redeploying, it’s safe to do it anyway — it only takes 10 seconds and won’t break anything.

KNOCK-ON EFFECTS Rolling back versions can break things if you don't also roll back the worker. Check CHANGELOG.md for the full list of what each version requires.

  • v2.1 → v2.0: Worker KV feedback routes still exist but are unused. Feedback falls back to localStorage. No harm.
  • v2.1 → v1.0: Worker must also be rolled back (git checkout v1.0 -- worker/ then npx wrangler deploy). White Label, DOCX, persistence, feedback all removed.
  • Any rollback: Articles in localStorage still load if the version supports them. Feedback in KV is preserved regardless.

Full version details

See CHANGELOG.md in the project root for detailed knock-on effects, verification steps, and worker rollback instructions for every version.
